# ADR-013: Multi-service architecture — SessionService, BlueprintService, OperatorService, BuildService, AuditService

## Context

ASHE's protocol surface had been described as a single capability-broker layer with multiple concerns (handshake, blueprint discovery, permission gating, audit, plus eventual build-mode operations). As the vision expanded through the May 2026 conversation arc to cover both **operator mode** (runtime decisions: read state, mutate state, query history) and **build mode** (code/system change proposals, evaluation, commit, rollback), the question arose of whether these should be one service or multiple.

The forces:

**Force 1: Build mode and operator mode have distinct lifecycles.** Operator mode is interactive — request, check, respond, audit. Build mode is transactional — propose, evaluate (potentially long-running with progress), commit or rollback. The lifecycle differences argue against bundling them into one service whose methods would have wildly different semantics.

**Force 2: Different consumers need different services.** An agent calling read-state operations needs OperatorService. An agent proposing code changes needs BuildService. An audit consumer needs AuditService streaming. Forcing them all through one service surface couples concerns that have different threat models, different rate-limit profiles, different consumer authorization.

**Force 3: Common infrastructure (session, blueprint, audit) is shared across all use cases.** Every mode needs identity validation, capability discovery, audit emission. These belong in shared services that the mode-specific services use.

**Force 4: gRPC's `service` keyword naturally supports multi-service partitioning.** A single .proto file can declare multiple services; clients import what they need; servers implement what they expose. This is the idiomatic gRPC pattern.

**Force 5: Deployment profiles (ADR-009) may include some services and exclude others.** An ASHE-core deployment might offer only SessionService + OperatorService; an ASHE-full deployment might offer all five. Service partitioning makes this clean (omit the service from the .proto registration) rather than messy (single service with feature flags per method).

## Decision

**ASHE's protocol surface is partitioned into five gRPC services declared in canonical `.proto` files:**

```protobuf
// ashe.proto — canonical service definitions

service SessionService {
  // Identity, session lifecycle, capability token issuance
  rpc Hello(HelloRequest) returns (Welcome);
  rpc Refresh(RefreshRequest) returns (Welcome);
  rpc Revoke(RevokeRequest) returns (RevokeResponse);
  rpc GetSessionInfo(SessionInfoRequest) returns (SessionInfo);
}

service BlueprintService {
  // System discovery — "what is this system, what can I do here"
  rpc GetBlueprint(BlueprintRequest) returns (stream BlueprintFragment);
  rpc GetAffordances(AffordanceRequest) returns (AffordanceCatalog);
  rpc GetCapabilities(CapabilityRequest) returns (CapabilityCatalog);
}

service OperatorService {
  // Runtime decisions and state interaction — the "operator mode"
  rpc CheckPermission(PermissionRequest) returns (PermissionResponse);
  rpc ReadState(StateRequest) returns (State);
  rpc MutateState(MutateRequest) returns (MutateResponse);
  rpc QueryHistory(HistoryQuery) returns (stream HistoryEvent);
  rpc StreamState(StreamStateRequest) returns (stream StateChange);
}

service BuildService {
  // Code/system change proposals — the "build mode"
  rpc ProposeChange(ChangeProposal) returns (ProposalReceipt);
  rpc EvaluateProposal(ProposalId) returns (stream EvaluationProgress);
  rpc CommitChange(CommitRequest) returns (CommitResponse);
  rpc RollbackChange(RollbackRequest) returns (RollbackResponse);
  rpc QueryProposalHistory(ProposalHistoryQuery) returns (stream ProposalEvent);
}

service AuditService {
  // Observability — capability exercise logging and forensics
  rpc StreamAudit(AuditSubscription) returns (stream AuditEvent);
  rpc QueryAudit(AuditQuery) returns (stream AuditRecord);
  rpc GetAuditDigest(DigestRequest) returns (AuditDigest);
}
```

**All five services share common infrastructure**:

- Identity model (every call carries a session token issued by SessionService)
- Scope grammar (capabilities scoped via the dotted-kebab-case convention)
- Error vocabulary (the 26 reserved error codes from PROTOCOL-SPEC §61 lineage)
- Audit pipeline (AuditService is the consumer of audit events generated by all services)
- Capability token format (issued by SessionService; verified by all others)
- Wire format (per ADR-012 — gRPC binary primary, JSON+TOON projections)

**Service partitioning is the public surface — internal implementation may share more.** Continuum's reference implementation may have substantial code-sharing across service handlers; the partition is what consumers see, not necessarily a strict implementation boundary.

**Deployment profiles (per ADR-009) determine which services are exposed**:

| Profile | Services exposed |
|---------|------------------|
| ASHE-core | SessionService + BlueprintService + OperatorService (basic) |
| ASHE-classify | + AuditService (basic) |
| ASHE-eval | + OperatorService (full) + AuditService (full) |
| ASHE-full | + BuildService + sandbox-mediated execution |
| ASHE-thin | SessionService + BlueprintService + OperatorService client; auxiliary services consumed remotely |

A deployment that doesn't expose BuildService simply doesn't register that service handler. Agents querying it via `BlueprintService.GetCapabilities` learn that build-mode capabilities aren't available.

## Consequences

**What becomes easier**:

- Clean separation of concerns at the protocol surface — each service has focused responsibilities
- Independent versioning per service — BuildService can evolve to v2 without forcing OperatorService through the same revision
- Selective deployment per profile — sites/kernels expose only the services they support; client codegen can target subset
- Authorization per service — capability scopes naturally map to "this scope authorizes calls into ServiceX"
- Conformance testing per service — separate test suites for each service; partial-conformance certification possible (e.g., "ASHE-core conformant" = passes SessionService + OperatorService basic tests)
- gRPC ecosystem benefits — service reflection, gRPC-Web for browser, gRPC-Gateway for REST projection all work per-service

**What becomes harder**:

- More artifacts to maintain — 5 service .proto files instead of 1 monolith (manageable; each is small and focused)
- Cross-service ordering may matter — calling BuildService.CommitChange before BuildService.EvaluateProposal has well-defined semantics that need documentation
- Capability scope partitioning needs care — scope names must clearly indicate which service they authorize (`construct.code.commit` for BuildService; `construct.composition.run` for OperatorService)

**What becomes possible**:

- Build mode and operator mode as first-class peers — not a feature flag on a single service; structural equality at the protocol level
- Future services can be added without disrupting existing ones — e.g., a future `OrchestrationService` for multi-agent coordination, a future `EconomyService` for capability-based monetization
- Implementations can choose which services to support based on their scope — a read-only data provider might expose only BlueprintService + OperatorService (read-only); a code review tool might expose only BuildService for code-evaluation operations
- Cross-implementation interoperability — a Rust ASHE server exposing BuildService can be consumed by a JS ASHE client wanting code-evaluation, regardless of either's other capabilities

**What becomes impossible** (intentionally):

- Hidden bundling of unrelated operations — every operation belongs to a clearly-named service
- "Capabilities depend on which method you call" obfuscation — services are explicit
- Monolithic "permission service" pattern that conflates session, scope, capability, and audit — ASHE's structure refuses this conflation

## Alternatives Considered

**1. Single ASHE service with all methods.** Rejected because:
- Doesn't reflect the distinct lifecycle of build mode vs operator mode (transactional vs interactive)
- Forces consumers to import the whole surface even when they need a subset
- Versioning becomes painful (whole service revs to add one method to one concern)
- Doesn't match gRPC ecosystem conventions where service partitioning is the norm

**2. Two services (CoreService + DomainService).** Rejected because:
- Build mode and operator mode are different enough that they don't belong in one DomainService
- Audit is cross-cutting; bundling it into either creates awkward coupling
- The five-service partition better matches the actual responsibility boundaries

**3. One service per resource type (UserService, ProductService, etc.).** Rejected because:
- ASHE is a capability-mediation protocol, not a resource API; resources are domain-specific concerns of the underlying system
- Conflates the protocol layer (ASHE) with the application layer (what the underlying system does)
- Would explode service count without architectural benefit

**4. RESTful resource model (no separate services).** Rejected because:
- REST's URI-as-resource model fits document/CRUD APIs poorly when the operations are inherently capability-mediated and intent-declared
- gRPC's service+method model maps more naturally to capability operations
- Streaming and bidirectional patterns (essential for AuditService and BuildService) are awkward in REST

**5. Microservices physically separate.** Rejected because:
- Five services don't need to run as five processes; they can be one process exposing five gRPC service handlers
- Physical separation is a deployment decision, not a protocol decision
- Operations may choose to deploy them separately (e.g., AuditService on dedicated hardware for compliance reasons); the protocol shouldn't dictate

## Implementation notes

- All five services declared in `infrastructure/protocols/ashe/schemas/ashe.proto` (canonical IDL)
- Shared message types (HelloRequest, CapabilityToken, EvaluatorResult, etc.) in `infrastructure/protocols/ashe/schemas/types.proto`
- Per-service detail .proto files possible if surface grows (e.g., `build-service.proto`); imported into the main `ashe.proto`
- Code generation produces typed clients for each service independently — consumers can import one without all
- Continuum's reference implementation: service handlers in `ashe-subsystems/services/` (one file per service)
